Website Performance Optimization: From 53 to 97 on PageSpeed Insights

This article is also available in: 中文.
Introduction

Recently, I performed a major overhaul of my blog, migrating the theme from the lightweight Cactus to the more feature-rich Icarus.

The cost of switching themes was painful: In previous optimizations, my site’s PageSpeed Insights score had reached 94. However, after switching to Icarus, due to the introduction of heavy JS, CSS, and external fonts, the mobile score plummeted overnight to 53.

Unwilling to accept this regression, I decided to perform a thorough “surgery” on this new theme. The goal was to maintain its rich features while achieving extreme loading speeds. After a series of targeted optimizations, the mobile score was finally pulled back to 97, almost touching a perfect score.

This article documents the complete optimization journey from refactoring to peak performance.

Initial Status Analysis

The PageSpeed Insights report highlighted the following major issues:

Metric Initial Value Issue
First Contentful Paint (FCP) 4.3s Too Slow
Largest Contentful Paint (LCP) 7.2s Severely Slow
Cumulative Layout Shift (CLS) 0.339 Significant Shift
Speed Index 5.2s Poor Experience

Key performance bottlenecks included:

  1. Oversized Font Files: MesloLG font TTF format was up to 636KB.
  2. Font Awesome CSS: Loaded the full icon library (~19KB) but only used a few icons.
  3. Missing Preconnect: No preconnect for critical resources like CDN.
  4. Unoptimized Images: Logo image was 447x432 but displayed at 88x88.
  5. Layout Shift: Images lacked width/height attributes causing CLS issues.

Why is removing Font Awesome so effective?

The core logic lies in the collapse of the Critical Request Chain.

Before: The browser had to go through a 4-level waterfall of HTML → style.css → all.min.css → fa-solid-900.woff2.
After: SVGs are inlined in the HTML byte stream. The request chain length drops from 4 to 0.

Round 3: SEO & Broken Link Fixes1. Eliminate /page/0/ 404 Links

Many Hexo themes generate <a href="/page/0/">Previous</a> on the first page, hidden by CSS but visible to crawlers. This causes 404 errors.

Solution: localized component rewrite (paginator.jsx) to conditionally render <span> instead of <a> when no valid page exists.

2. Remove Blocking Progress Bar (Pace.js)

pace.min.js was a blocking resource in <head> that contributed nothing to LCP. I disabled it in _config.icarus.yml.

  2. Legacy Formats: Still using JPG/PNG instead of WebP.

Round 5: Eliminating Critical Request Chain Blocking

PageSpeed Insights still flagged a 441ms “Critical Request Chain” block, caused by fonts.googleapis.com.

Actions
  1. Remove Google Fonts: Removed the default Google Fonts loading in the theme. Since we configured local fonts in Round 1, this external request was redundant. Saved 441ms.
  2. Async Load Non-Critical CSS: Changed highlight.js styles (atom-one-light.css) to load asynchronously.

This reduced the critical request chain depth from 5 to 2.

Lessons Learned
  1. Fonts Matter: Web fonts are often overlooked performance killers; WOFF2 is preferred.
  2. Use Icons Wisely: Inline SVG is far more efficient than loading a full library for just a few icons.
  3. Images on Demand: Ensure image dimensions match display dimensions (e.g., 672px/600px).
  4. Critical Chain: Remove blocking external resources (like Google Fonts) from the critical path.
  5. CLS is Fixable: Add explicit width/height to images.
  1. Cache TTL: Use CDN rules to force long-term caching (max-age=31536000) for static assets.
